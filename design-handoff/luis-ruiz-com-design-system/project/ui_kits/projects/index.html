Nucleus: a bring-your-own-key API product on top of a portfolio site.

Bearer-token auth, atomic credit accounting, and provider-backed chat — packaged as a paid API surface sharing one Supabase backend with the rest of the site.

Next.js 15 App Router Supabase + RLS Stripe OpenAI Anthropic JOSE / JWT Upstash rate-limit
Context

Why I built it

luis-ruiz.com was already running multiple AI experiments in public — local Ollama chat, a multi-model round-robin tool, embeddings playground. The next honest step was packaging one of them as a real product: an API surface that gives a developer access to multiple model providers under a single bearer token, with credits and usage tracking instead of a sprawling SaaS dashboard.

Nucleus is not a portfolio piece. It's a real product surface that lives inside the same repo as the rest of the site. The goal was to prove the brand could run one full-stack runtime for both marketing and a paid API without either side blurring into the other.

Problem

Single bearer token, many models, no leaks

Routing a chat call through three providers (OpenAI / Anthropic / xAI) while staying inside one auth pattern means three things have to hold together at once:

  • A single Supabase user can mint a Nucleus bearer JWT that survives provider-side rate limits.
  • Credits deduct atomically — never double-charge a retried call.
  • Pro subscribers get unlimited access without breaking the credit ledger.
Constraint: Nucleus had to share infra with the public site without ever exposing service-role keys through the marketing routes.
Approach

Thin route handlers, dense service layer

API routes under app/api/nucleus/* stay thin: validate, call into lib/nucleus/*, return normalized JSON. The interesting logic lives in:

  • lib/nucleus/auth.ts — bearer JWT verify against Supabase signing key.
  • lib/nucleus/credits.ts — RPC-backed atomic deduct / add via Postgres functions.
  • lib/nucleus/pricing.ts — model catalog + per-token pricing config.
  • lib/nucleus/rate-limit.ts — Upstash sliding window per key.
